01 · Mental model

The 60-second mental model

You don't memorise stages. You memorise which prompt to use when.

Inbox① capture
Backlog② triage
Ready③ DoR passed
Active④ in-progress
Review⑤ in-review
Done⑥ DoD passed
TL;DR
There are 10 prompts in 92-Prompts/. You'll use four of them 80% of the time. The other six are scaffolding for less-frequent moves (Epic drafting, OKR refresh, monthly retro).
02 · Prompts

The four prompts you live in

Memorise these four. Open the kit's 92-Prompts/ folder for the full text.

05
Friday · PM hat

refine-backlog-to-ready

Promote 5 Inbox/Backlog items to Ready by running the DoR checklist on each. Weekly.

06
Start of session · Dev hat

execute-story

Pull a Ready story. Claude codes, makes decisions on the spot as ADRs, flips to in-review. Daily.

07
After code done · QA hat

run-testplan

Run every TC command. PASS/FAIL each. Auto-file bugs on failure. Daily.

08
After tests pass · QA→PM

close-out-story

Run DoD checklist. Flip to done. Update MONITOR + dashboard. Daily.

The other six prompts
01 draft OKRs (quarterly) · 02 draft Epic (when a new strategic bet appears) · 03 split Epic into Features (after 02) · 04 split Feature into Stories + Testplans (after 03) · 09 weekly monitor update (Friday) · 10 monthly retro (1st Monday).
03 · Worked example

End-to-end: "add a wishlist feature"

One new requirement, all six steps, exactly what you type at each one.

  1. Capture

    30 secany hat

    You're in the middle of something else. An idea hits. Open Claude:

    Capture this in the Inbox: customers can save items to a wishlist.
    Creates 10-Inbox/INBOX-NNNN-customer-wishlist.md with status: not-started. You go back to what you were doing.

  2. Friday refinement

    30 minPM hat

    Friday afternoon. Paste prompt 05 (refine-backlog-to-ready.md).

    Lists 5 unrefined items. For each, asks: "What OKR does this serve? What's the acceptance criteria?" If approved, promotes:
    • Writes a new Epic (or links to existing)
    • Drafts Features under it
    • Drafts Stories with paired Testplans

    Each new artefact starts not-started; must pass DoR to become ready.

  3. Monday: pull one to work

    10 minDev hat

    Open Claude. Quick orientation:

    What's ready to pull?
    Reads DASHBOARD.html, lists Ready stories. You pick one — say STORY-12.3.04-wishlist-add-button.

    Then paste prompt 06 with the story path.

    • Verifies DoR is met
    • Flips status: in-progress, sets started_at
    • Reads existing patterns from your codebase, writes the component
    • Hits a decision (e.g. "Firestore subcollection or separate collection?") → STOPS, creates ADR-NNNN-wishlist-storage.md, picks an option, continues
    • Ticks AC checkboxes as completed

    2–3 hours later: code done. Claude flips to in-review.

  4. Verify

    15 minQA hat

    Same day or next morning. Paste prompt 07 with the testplan path.

    • Runs every TC's Command: against your actual code
    • Updates each TC's Result: line with PASS or FAIL
    • On any FAIL → automatically files BUG-YYYYMMDD-NN-*.md with repro steps + first analysis + fix direction
    • All PASS → reports green, recommends prompt 08

  5. Close out

    5 minQA → PM

    Paste prompt 08.

    Runs full DoD checklist (lint, tests, build, ADR present, MONITOR updated, dashboard regenerated). If anything fails — stops and asks. If all pass:
    • Flips status: done, sets completed_at
    • Updates 42-Monitor/MONITOR.md revision history
    • Runs npm run pm:dash

    Refresh DASHBOARD.html in your browser — wishlist story green, metrics tick up.

  6. Friday again

    30 minPM hat

    Paste prompt 09 (weekly-monitor-update.md).

    Summarises the week, updates MONITOR with revision history, regenerates dashboard, flags stalled stories.

05 · Surfaces

Where you "live" day-to-day

Two surfaces. Almost never anywhere else.

Input surface

Claude Code session

Where you paste prompts. Claude reads/writes markdown, runs tests, regenerates the dashboard.

Output surface

DASHBOARD.html (browser tab)

Live view of every artefact. Click rows to see body. Always one refresh away from current state.

Why this matters
You almost never open a markdown file directly. Claude reads and writes them; the dashboard renders them. The markdown is the storage; the dashboard is the interface.
06 · Sizing

Sizing a new requirement

When something new comes in, decide its size. That determines which prompt to use.

One-line idea
→ Inbox · single sentence to Claude
(no prompt)
1–2 hour fix
→ Backlog → straight to Ready in next Friday refinement
prompt 05
1–2 day feature
→ New Story under existing Feature (with paired testplan)
prompt 04
1–2 week feature
→ New Feature (and its stories) under existing Epic
03 → 04
Strategic bet
→ New Epic, with features and stories
02 → 03 → 04
Don't know the size? Ask Claude.
I want customers to be able to save items. Where does this fit — Story, Feature, or Epic? Propose the breakdown.

Claude reads your existing Epics, OKRs, relevant code, and proposes the right level. You approve or refine. That's the entire planning ceremony.

07 · Bootstrap

The first session after deployment

Your very first time using the kit on a fresh project. ~3 hours start to first story pulled.

  1. Open the dashboard

    1 min

    Open 42-Monitor/DASHBOARD.html. It shows "0 artefacts". Correct.

  2. Write the North Star

    30 minFounder hat

    Open 00-Strategy/NORTH-STAR.md or use prompt 01. One page.

  3. Write the OKRs

    30 minFounder hat

    Use prompt 01. Three Objectives, 2–3 KRs each.

  4. Draft the first Epic

    45 minPM hat

    Use prompt 02. Pick the highest-impact OKR. Let Claude draft.

  5. Split into Features

    30 minPM hat

    Use prompt 03 on the Epic.

  6. Split first Feature into Stories

    45 minPM hat

    Use prompt 04. Now you have Stories + paired Testplans.

  7. Refine top story to Ready

    10 minPM hat

    Use prompt 05 on that one story.

  8. Pull it to work

    go!Dev hat

    Use prompt 06. You're now operating the kit.

08 · Edge cases

When a prompt doesn't fit

The 10 prompts cover ~95% of moves. The other 5%:

Edge case

Spike / exploration

No formal artefact yet — research only. Use a BACKLOG-NNNN-spike-*.md to capture the question + findings. Promote to a Story only when the spike answers something concrete.

Edge case

Hotfix in production

Bypass the lifecycle. Fix the code. AFTER deploy, retroactively create a BUG file + a Story with status: done describing what was done. Don't skip — future Claude needs the trail.

Edge case

Refactor with no user-visible outcome

File as a Backlog entry (type_of_work: tech-debt). Promote to a Story when scoped enough to estimate.

Edge case

Documentation-only work

Story with type_of_work: docs. Testplan can be minimal — a TC that greps for the expected content.

09 · Hats

Hat protocol — when to announce

At the start of any substantive Claude session, state which hat. Mixing hats is the most common cause of scope drift solo.

★
Strategy

Founder hat

Strategic decisions, OKRs, epic approvals, sunset calls.

◇
Planning

PM hat

Refining backlog, drafting epics/features/stories, MONITOR updates.

▲
Execution

Dev hat

Executing stories, writing code, making implementation decisions (ADRs).

●
Quality

QA hat

Running testplans, filing bugs, signing off DoD.

Why announce
It tells Claude which mode to optimise for. PM hat asks "is this scoped?". Dev hat asks "is this efficient?". They give different answers to the same question.
10 · Anti-patterns

Six ways to break the system

Avoid these.

① Pulling work that isn't Ready

If a story hasn't passed DoR, paste prompt 05 first. Don't skip.

② Closing without DoD

"Tests are flaky so I'll skip them" → file a BUG instead. Leave story in-review.

③ Skipping ADRs

Three months from now you won't remember why you picked option A. Future-Claude won't either.

④ Letting MONITOR drift

If the dashboard hasn't been regenerated in a week, you've lost the operating signal. Run npm run pm:all every Friday minimum.

⑤ Hoarding Inbox items

If something has been not-started for 90 days, archive it. Sunset rule (SOP §15).

⑥ Re-deriving conventions in chat

If you find yourself explaining the same thing to Claude twice, it belongs in PROJECT-CONTEXT.md.
