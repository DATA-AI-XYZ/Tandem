HTML output convention

The kit renders its human-facing views from the same Markdown that holds the plan. The rule is uniform: a source *.md renders to a sibling *.html of the same basename, the HTML is a generated artefact, and you never hand-edit it. This page documents that convention so the read views stay reproducible.

The rule

  • Source is Markdown. Each documentation page is a documentation/*.md file. The first

    • # heading becomes the page title and the Docs-view tile name.

  • Output is a sibling .html. npm run pm:docs reads every documentation/*.md and

    • writes a self-contained *.html beside it — architecture.md → architecture.html. The HTML inlines the kit's brand tokens, so it needs no external CSS.

  • Generated, never hand-edited. The .html is overwritten on every render. Edit the

    • .md and re-run the generator; any manual change to the .html is lost on the next build.

  • Idempotent. Same input produces the same body HTML — the generated-at stamp lives only

    • in an HTML comment, so re-rendering an unchanged page yields an identical styled body.

Where the convention applies

The same *.md → *.html pattern runs across the kit's generated surfaces:

  • Documentation — generate-docs.js (npm run pm:docs) renders this documentation/

    • folder. buildDocs() in the dashboard generator then lists every documentation/*.html as a Docs-view tile, so a new page appears simply by adding its .md and re-rendering.

  • Dashboard — generate-dashboard.js (npm run pm:dash) renders the whole board to

    • 42-Monitor/DASHBOARD.html from the plan's Markdown artefacts.

  • Release — the published build ships the rendered *.html pages into the public

    • docs/ site, so the same sources serve both the in-repo Command Center and the public docs.

Practical notes

  • To add a page: create documentation/<name>.md, give it a # title and at least a couple

    • of ## sections, then run npm run pm:docs.

  • To remove a page: delete both its .md and its .html, or the stale tile lingers in

    • the Docs view (which lists the rendered .html).

  • Do not commit a hand-edited .html: regenerate it instead, so the source and the output

    • never disagree.
