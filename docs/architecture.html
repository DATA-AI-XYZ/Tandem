Architecture

Tandem is a Claude Code plugin that wraps a repo-native, file-based project-management system. There is no server, no database, and no runtime service: the "application" is a folder of Markdown artefacts, a set of slash-command skills that read and write those artefacts under fixed rules, and a handful of stdlib-only Node scripts that lint the folder and render it to HTML. This page explains the layers that make up the kit and how project-management data flows through them.

Layers

The kit is built from five cooperating layers. Each owns one concern and talks to the others through plain files on disk — never through shared memory or a network call.

  • Plugin skills (the command surface). Each lifecycle step (draft-prd,

    • split-into-features, execute-batch, close-phase, …) is a skill that loads its own instructions on demand and acts as a single PM "hat" (Founder, PM, Dev, QA, Technical Writer). Skills are the only thing that create or mutate artefacts; they encode the gates (Definition of Ready, Definition of Done) rather than leaving them to memory. Skills register by auto-discovery — placing a SKILL.md in skills/ is sufficient, and each pins an explicit name equal to its folder (ADR-0042).

  • Sub-agents. Read-heavy or noisy work is delegated to a fresh sub-agent: an Explore agent

    • for "where is X / which files reference Y", a discipline sub-agent (technical-writer, frontend-developer, …) for focused authoring. Sub-agents return evidence; the main thread keeps the editing, the decisions, and the status flips. A sub-agent never owns a status change.

  • The PM tree (the artefact store). The artefact store lives under _00-Project-Management/ — folders that

    • hold one Markdown file per artefact (epics, features, stories, testplans, bugs, decisions, monitor, reports, standards, templates, scripts). Frontmatter carries the contract: a closed-set status: enum, ISO-8601 timestamps, and the relationship links (epic:, feature:, story:, testplan:) that stitch the tree together. Folder *roles* are resolved through a logical→physical path map (90-Standards/pm-paths.json, ADR-0052) with full and flattened presets, so the kit works for projects that do not use the canonical numbering.

  • The tooling. The scripts under _00-Project-Management/93-Scripts/ are the kit's

    • "build": validate-frontmatter.js (pm:lint) enforces the frontmatter contract, generate-monitor.js (pm:monitor) recomputes the board rollups, generate-dashboard.js (pm:dash) renders the interactive Command Center, and generate-docs.js (pm:docs) renders this documentation/ folder. install.js (pm:install) and update.js (pm:update) materialize and refresh the PM tree from a manifest; batch-check.js (pm:batch-check) proves a batch of stories touch disjoint files before parallel execution; doctor.js (pm:doctor) reports wiring health. The validator and generators share one frontmatter-parser module so they read the contract identically (ADR-0067). All are dependency-free Node stdlib and idempotent.

  • Hooks. Two Claude Code hooks make enforcement deterministic rather than discretionary,

    • both dispatched through a single canonical Node entrypoint, 93-Scripts/hook.js (ADR-0053), via ${CLAUDE_PLUGIN_ROOT} so they run identically on Windows, macOS, and Linux: a PostToolUse hook lints frontmatter after any edit under the PM folder, and a Stop hook regenerates the dashboard when PM files changed.

PM data flow

A unit of work flows down the artefact tree and back up the board. The data moves strictly through files; each step is gated before the next can begin.

  1. Strategy → Epic. A North Star and quarterly OKRs in 00-Strategy/ justify an

    2. EPIC-NN. Every epic must link to strategy or it is rejected — work has to trace to a business outcome.

  2. Epic → Features → Stories + Testplans. split-into-features decomposes an epic into

    3. FEAT-NN.M files; split-into-stories decomposes each feature into STORY-NN.M.PP files and writes the paired TESTPLAN-NN.M.PP in the same step. Pairing is mandatory: every acceptance criterion maps to at least one runnable test case.

  3. Refine → Ready. refine-backlog runs the Definition-of-Ready checklist and flips a

    4. story to ready. Only ready stories may enter execution.

  4. Execute → Verify → Close. execute-story (or execute-batch for a grouped "chat")

    5. flips ready → in-progress, implements the acceptance criteria one at a time, and files ADRs and BUGs as they arise. A file-disjoint batch can instead run through execute-batch-parallel, which fans out one sub-agent per story and folds the results back through a single-writer reconciliation — the main thread stays the sole writer of the board and the sole assigner of sequential ADR/BUG IDs (ADR-0075). run-testplan runs each test case's command verbatim and auto-raises a BUG on any failure. close-out-story runs the Definition-of-Done gate (it includes an AI-code review) and flips the story to done.

  5. Board update. Every status change writes back to 42-Monitor/MONITOR.md (the counts,

    6. the progress bars, a revision-history line) and regenerates DASHBOARD.html. The dashboard and these docs are read views; the Markdown files remain the single source of truth.

Canonical assets & the manifest installer

The kit keeps one copy of the PM system. The working _00-Project-Management/ tree is the canonical source *and* the board this project is run from — the kit dogfoods itself on its own tree. There is no separate scaffold/ template to keep in sync: the earlier live-vs-scaffold split was retired in favour of shipping the canonical assets directly and materializing them in each target (ADR-0074). A single lib/pm-manifest.json lists the work-folders and seed files, each tagged with an ownership boundary — kit-owned files are overwritten on install and update, user files are written only when absent (ADR-0072). install.js builds a target's tree from that manifest; update.js refreshes only the kit-owned entries, so pulling a kit upgrade never disturbs your work (ADR-0073).

The public release pipeline

The kit develops in a private repository and ships a sanitized public build (ADR-0028). build:tandem allowlist-copies the product surface (the canonical PM assets plus the 93-Scripts/ tooling and templates the installer needs), rewrites the project-name and contact tokens, renders a sanitized demo board, and runs a scrub gate that fails the build if any internal-provenance token survives — an inverted binary-blocklist with a NUL-byte sniff, a case-insensitive denylist, and a fail-closed symlink rejection (ADR-0040). A fresh install of the built plugin then materializes a working PM tree from the shipped manifest. Because the public bundle ships documentation/ but not _00-Project-Management/, any shipped doc references decision records by ID in prose (for example "ADR-0028") rather than relative-linking into the non-shipped PM tree.

Why-it-is-this-way

Decisions worth remembering are captured on the spot as ADR-NNNN files under 40-Decisions/ and linked from the story that triggered them — so the *why* behind a field name, a threshold, or a deviation survives long after the commit that made it. See Decisions for the full digest.
