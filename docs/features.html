Tandem · PM Operating Kit

Features

This page describes what Tandem can do, grouped by capability. Each capability is delivered by one or more skills (slash commands Claude loads on demand) and/or scripts (npm run pm:*). Everything below is shipped — the kit is at v2.4.0 with all planned epics complete. For the exact order you run the lifecycle skills in, see the Command lifecycle; for the underlying model, see Architecture.

The full feature tree lives under _00-Project-Management/31-Features/ (50 feature files across 14 epics). The sections here summarise the major groups rather than listing every file.

Strategy & planning

Anchor work to a business outcome before any code is written.

  • draft-okrs (Founder hat) — draft quarterly OKRs from a North Star.
  • draft-prd (Founder → PM hat) — draft a PRD from an OKR or rough notes; ships its own

    • PRD template.

  • draft-epic (PM hat) — draft an Epic from an OKR key result or a PRD section. Every

    • epic must link back to strategy or it is rejected.

Decomposition

Break strategy down into executable, testable units.

  • split-into-features (PM hat) — decompose an Epic into FEAT-NN.M files.
  • split-into-stories (PM hat) — decompose a Feature into STORY-NN.M.PP files and

    • their paired TESTPLAN-NN.M.PP in the same step. Story-to-testplan pairing is mandatory: one test case per acceptance criterion.

Refinement & gates

Two checklists bracket execution so only ready work starts and only finished work closes.

  • refine-backlog (PM hat) — run the Definition-of-Ready checklist and either promote a

    • story to ready or stop and list the gaps. It never silently promotes.

  • close-out-story (QA → PM hat) — run the Definition-of-Done gate (which includes an

    • AI-code review), flip the story to done, update the board, and regenerate the dashboard.

Execution

Pull ready work into progress — one story or a planned batch.

  • execute-story (Dev hat) — start a single ready story: verify Definition of Ready, flip

    • to in-progress, work the acceptance criteria one at a time, filing ADRs and BUGs as they arise.

  • run-testplan (QA hat) — run every test case's command verbatim, mark PASS/FAIL, and

    • auto-file a BUG for any failure before reporting.

  • execute-batch (Dev/QA hat) — execute one "chat" from an Implementation Strategy: run

    • its stories sequentially (execute → test → close) with atomic per-story finalisation and a context-budget guard.

Intelligent execution strategy

Plan *how* to execute a backlog, not just what.

  • execution-strategist (PM hat) — group ready stories into right-sized "chats" by

    • dependency and coherence (judgment-led grouping, soft size bounds; ADR-0025/0026). It is a dry-run planner — it does not change story status.

  • Story sub-agent metadata — each story carries a type_of_work (and optional

    • suggested_agents) used to pick a specialist executor, with a documented fallback so a missing agent never hard-fails (ADR-0023).

Review & critique

Two complementary review surfaces.

  • critique (PM hat) — advisory quality review of a planning artefact (Epic, Feature,

    • Story, Testplan). Read-only; it never rewrites the file.

  • peer-review (reviewer / QA hat) — on-demand code review of an explicit target

    • (diff, branch, PR, or file) across correctness, security, performance, maintainability, tests, and error paths, returning blocker/major/minor findings each with a suggested fix.

Phase lifecycle

Group a set of stories into a phase, run it on its own branch, and integrate cleanly.

  • start-phase (PM hat) — gate the entry state, cut a phase/<phase-id> branch off

    • main (ADR-0045), and record the open on the board.

  • close-phase (PM hat) — gate on every phase story being done, compile a phase

    • retrospective, capture follow-ups, update the board, and run a gated merge to main.

The Command Center

npm run pm:dash renders the whole board to an interactive 42-Monitor/DASHBOARD.html: status pills, progress bars, and a workflow-aligned, grouped information architecture built on a shared tile-grid + drawer component model (ADR-0032/0033/0048). It includes an Implementation Strategy view (each epic's strategy selected by story coverage, ADR-0049), an AI catalogue that describes the installed skills/agents for humans, a Docs view that serves these pages, and a context-cost view that surfaces the token "tax" of what each session loads. A zero-dependency headless smoke (pm:smoke, ADR-0038) drives system Chrome over the DevTools Protocol to verify the render with no Puppeteer/Playwright.

Founder-facing outcomes

Every planning artefact carries a plain-English "what you'll be able to do" line so the plan reads in business terms.

  • write-outcomes (dispatch-only) — a shared skill the producer skills spawn via a

    • sub-agent to turn an artefact's technical scope into one founder-outcome line (ADR-0059).

  • Outcomes are authored automatically across the planning tree (PRD → feature → story →

    • chat/phase strategy), with a refine-backlog fill-if-missing safety net, and rendered on the board as the chat-card lead, the phase subtitle, and in the artefact drawer.

Cross-project portability

The kit adapts to a host project rather than assuming one layout.

  • Configurable folder layout — PM sub-folders resolve through a logical→physical path map

    • (90-Standards/pm-paths.json, ADR-0052) with full and flattened presets plus per-key overrides, so the generators and validator work even when a project does not use the canonical numbering.

  • pm:install / pm:doctor — one idempotent command to wire every pm:* script and pin

    • the layout; a health check to verify it (ADR-0054). The installer registers the hooks into the target only when none exist (a guarded write, ADR-0055).

  • Cross-platform hooks — a single canonical Node entrypoint runs the hooks identically on

    • Windows, macOS, and Linux (ADR-0053), with no bash/jq dependency.

  • Tiered CLAUDE.md model — fill-claude-md and claude-audit enforce a lean,

    • reference-don't-duplicate CLAUDE.md layout with advisory line budgets.

Stack-aware tooling

The kit is stack-agnostic — it ships per-type guidance for web-app, mobile, CLI, library, backend-service, data-pipeline, Power Platform, and automation projects. curate-toolkit (PM hat) ranks the installed AI tools (skills, agents, commands, plugins) HIGH / MED / LOW by fit for the project type and writes relevance overlays, so the AI catalogue reflects what is actually useful for *this* project.

Documentation generation

document (Technical Writer hat) authors this default documentation set — overview, getting-started, architecture, decisions, and features — synthesised from the project's own PM knowledge (PROJECT-CONTEXT, epics/features/stories, ADRs, and the codebase). It authors Markdown only; npm run pm:docs renders the folder to HTML, which the release pipeline ships into the public docs/ site (ADR-0030/0034).

Quality tooling & the release pipeline

  • pm:lint — validate the frontmatter contract across every artefact, with a non-fatal

    • warning tier separate from hard violations (ADR-0061).

  • pm:monitor — recompute the board counts and rollups from real story statuses

    • (ADR-0024); pm:mirror guards the live-vs-scaffold split against drift.

  • build:tandem — produce the sanitized public release behind a scrub gate that fails on

    • any internal-provenance token (ADR-0028/0040).

Session helpers

  • session-start — orient at the start of a session: read the active WIP, recent ADRs, and

    • the board, then announce the suggested next step.

  • weekly-monitor / monthly-retro — cadence reviews that update the board and

    • capture retrospectives.

  • reflect — end-of-session reflection that proposes CLAUDE.md / SOP / skill updates for

    • you to approve.
