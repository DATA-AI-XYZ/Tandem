Context economics

Every skill, sub-agent, and bundled plugin you install costs something *before it does any work*: it occupies space in the model's context window the moment it loads. This page defines that cost — the kit calls it the context tax — and explains the token-cost view (delivered in FEAT-11.3) that lets you see and manage it.

What the context tax is

Context-load cost is the number of tokens an artefact adds to the context window simply by being loaded — its instructions, its always-loaded references, the definition the model must read before it can act. The kit names this the context tax: a standing charge you pay on every session that loads the artefact, whether or not you ever invoke it.

It is not a completion cost. It is explicitly distinct from the per-invocation completion cost — the tokens spent generating a response when a command actually runs — and it is not a dollar cost. The figure is a *context-weight* estimate measured in tokens, not money and not a per-invocation charge. The point of the metric is capacity, not billing: a large catalogue of rarely-used skills can quietly consume the window and crowd out the work you actually care about, and the context tax is how you notice that before it bites.

Why it matters:

  • The context window is finite; the tax is paid up front, every session.
  • A cheap-to-invoke artefact can still be expensive to *load* if its instructions are long.
  • Trimming or deferring high-tax artefacts you rarely use buys back working space.

Reading the token-cost view

FEAT-11.3 surfaces the context tax directly in the AI catalogue (the Command Center's toolkit view in the dashboard). The token-cost view counts the tax per scope:

  • Command / skill — the tokens of the skill body (its SKILL.md) plus any references it

    • always loads.

  • Sub-agent — the tokens of the agent definition the model reads to adopt that role.
  • Plugin total — the summed cost of everything a bundled plugin auto-loads, shown as a

    • distinct plugin total so a whole plugin's footprint is visible at a glance.

How to read it:

  • Per-card figure — each catalogue card shows its own ~N tok context-load badge.
  • Per-kind rollup — a roll-up header totals the tax by kind (commands, skills,

    • sub-agents), so you can see where the weight concentrates.

  • Plugin total — the bundled-plugin total is reported separately and deliberately kept

    • out of the grand total, to avoid double-counting folded-in items.

  • Sort and filter by cost — sort the catalogue ascending/descending by context-load

    • cost, and filter to a minimum cost, to find the heaviest artefacts fast.

Read the number as *context weight*, not price. For the precise counting method behind the figure, see FEAT-11.3's decision record; this page defines what the number means and how to use the view to keep your context economics healthy.
