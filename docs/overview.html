Overview

Tandem · PM Operating Kit

Overview

Tandem is your Claude Code co-pilot for shipping ideas without the chaos. It takes you from idea to production and makes sure nothing slips: no story goes in-progress without a testplan, no work ships without passing its gates, no decision disappears into the chat log. The result is a team-quality delivery rhythm, at solo-founder pace.

Most "AI project management" is a chat log. Tandem is a contract. Under the hood it is a repo-native PM operating kit: the plan lives as plain Markdown under _00-Project-Management/, a set of slash-command skills create and advance those files under fixed gates, and a few stdlib-only Node scripts lint the plan and render it to an interactive Command Center and to these documentation pages. There is no app to deploy and no database to run.

This page is the orientation. For how the pieces fit together see Architecture; to get a project running see Getting started; for the full capability list see Features; for the record of why the kit is built the way it is see Decisions.

Who it is for

A solo founder (or a small team) who works *with* Claude Code and wants the rigour of a real PM process without the overhead of a separate tracker. The kit is opinionated on purpose: it encodes one coherent way to go from a North Star down to shipped, verified work, and it makes Claude follow that way every session rather than improvising a new one each time.

What you get

  • A numbered artefact store. Epics, features, stories, testplans, bugs, and decision

    • records each live in their own numbered folder, one Markdown file per artefact, stitched together by frontmatter links. The Markdown is the single source of truth.

  • Lifecycle slash-command skills. Every step from drafting a PRD to closing a phase is a

    • skill that loads its own instructions on demand and acts as one PM "hat" (Founder, PM, Dev, QA, Technical Writer). The skills enforce the gates so you do not have to remember them.

  • A live Command Center. npm run pm:dash renders the whole board to an interactive

    • Command Center dashboard (local in-repo only) — status pills, progress bars, an Implementation Strategy view, an AI catalogue, and these docs.

  • Quality gates as scripts. npm run pm:lint validates the frontmatter contract;

    • npm run pm:monitor recomputes the board counts from the real story statuses; a mirror check guards the live-vs-template split from drift.

  • Founder-facing outcomes. Each planning artefact carries a plain-English "what you'll be

    • able to do" line, authored automatically and surfaced on the board, so the plan reads in business terms and not just technical scope.

How the work flows

Strategy anchors everything. A North Star and quarterly OKRs justify each Epic; an epic is split into Features; each feature is split into Stories, and every story is created with its paired Testplan in the same step. A story is refined to ready through a Definition of Ready checklist, executed (alone or in a grouped "chat" batch), verified by running its testplan, and closed through a Definition of Done gate. Every status change writes back to the board. See the Command lifecycle for the exact order.

Core rules to know

  • Status is a closed set of nine values — never invent new ones.
  • Stories never move folders — only their status: field changes.
  • Every story has a paired testplan — created together, one test case per acceptance

    • criterion.

  • Decisions are recorded on the spot as ADR-NNNN files, linked from the story.
  • A defect raises a BUG immediately, before it is reported anywhere else.

Status

Stable and actively maintained — stack-agnostic (web, mobile, CLI, library, backend, data-pipeline, Power Platform, and automation project types), with the full North Star → Done lifecycle shipped across 24 skills, a hardened release pipeline, and cross-project portability. Tandem dogfoods itself: this very documentation set is generated from Tandem's own board.
